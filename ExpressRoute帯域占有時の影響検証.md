# ExpressRoute帯域占有時の影響検証

## 概要

本検証では、Microsoft AzureとOracle Cloud Infrastructure（OCI）をExpressRouteおよびFastConnectによる閉域接続で接続した環境において、  

**ExpressRoute回線の帯域を占有した場合にどのような影響が発生するか**  

について、Azure環境の観点から検証を実施した。

検証観点は以下の通り。

- ExpressRoute回線の帯域占有時における他通信への影響
- 帯域ひっ迫の検知方法
- Azure環境における通信の強制停止方法

### ここから考えられること
- データをパケットとして分解して送信する場合、分解する数が多くなるため、転送に時間がかかるのは当然であるといえる。
- 仮想マシンのNICそのものの性能や物的な回線などがボトルネックになるため、帯域使用率はファイル容量に比例して大きくなるのものではない。
- ファイルの容量は、帯域占有の**発生する時間**に影響を与える

## アプローチを変えて、確実に帯域を占有させる
OCI環境にもう一台インスタンスを立て、iperf3でAzureVMから継続的に通信を実行し、帯域を占有する。<br>
その間、もう1台のAzureVMからファイルを共有フォルダにコピーする

![alt text](</img/スクリーンショット 2026-03-01 070208.png>)

<br>

OCI環境にUbuntuのインスタンスを立てる。<br>
※　インスタンス作成方法は、Windowsのインスタンスと同じ

![alt text](</img/スクリーンショット 2026-03-01 065309.png>)

<br>

ローカル環境PowerShellを立ち上げ、新しいUbuntuのインスタンスにSSH接続し、<br>
下記コマンドでiperf3をインストールする

**Ubuntu(OCI)側の実行コマンド** 

``` Bash
# sudo権限推奨
sudo apt update
sudo apt install -y iperf3

sudo iptables -I INPUT 1 -p tcp --dport 5201 -j ACCEPT

# サーバ起動、iperf3サーバーとして待ち受け状態となる
iperf3 -s

```

![alt text](</img/スクリーンショット 2026-02-28 234914.png>)

<br>

続いて、AzureVM1から実行するiperfのコマンドをインストールする

**Windows(Azure)側VM1の実行コマンド**
``` Powershell
Set-ExecutionPolicy Bypass -Scope Process -Force
[System.Net.ServicePointManager]::SecurityProtocol = [System.Net.ServicePointManager]::SecurityProtocol -bor 3072
iex ((New-Object System.Net.WebClient).DownloadString('https://community.chocolatey.org/install.ps1'))

choco install iperf3 -y
```

<br>

下記を実行すると、16個の通信セッションが開始される。

```
iperf3 -c 132.145.127.67 -P 16 -t 0
```

> ℹ️ iperf3とは<br>
>ネットワークのパフォーマンス測定を行うためのツールです。<br>
>これを使って、以下のことを行います。
>
>- 16個のTCPセッションを同時に開く
>    - → 単一セッションよりもTCPのスループットが高くなる
>    - → ExpressRoute 1Gbps をフルに使いやすい
>    - → 必要に応じて、-Pの数字を増量する（特に上限はない）
>- 各ストリームで データを送り続ける
>    - → 帯域を飽和させ、ネットワークが常に100％近くになる
>- 無限送信（-t 0）なので、停止するまでずっと送信し続ける
>    - → この状態で別のVMからファイルを転送すると、帯域が圧迫されて欠損や再送が発生するかを確認できる


![alt text](</img/スクリーンショット 2026-03-01 000406.png>)

<br>

一気に回線帯域使用率が臨界に到達した

![alt text](</img/スクリーンショット 2026-03-01 002721.png>)

<br>

また、この時、パケットのドロップも発生していた

![alt text](</img/スクリーンショット 2026-03-01 050037.png>)

<br>

この状態でAzureのVM2から、ファイル転送(1GB×40個)を実行。<br>

### 結果

- 帯域を大容量通信で占有していても、**他の通信が完全に停止するわけではない**ことを確認した。<br>
- また、帯域使用率は仮想マシンの通信性能などにも左右されるため、一概にファイルの容量だけで左右されるものではない。<br>

    - ※本検証において、わずかに100%から下振れが瞬間的に生じるため、ファイルの欠損が起きず速度にもさほど影響はないと考えられる。

## 通信を止める方法の検証

### 1. NSGで全拒否ルールを追加

送信元サブネットのNSGにて、最優先ルールで「すべての通信をDeny」に設定。<br>
しかし、既存セッションは継続し、アップロードは停止しなかった。

![alt text](</img/スクリーンショット 2026-02-14 134540.png>)

<br>

---

### 2. ルートテーブルの無効化

defaultサブネットに対し、  空のルートテーブルを関連付けた。

<br>

結果として、

- 宛先へのルートが見つからないエラーが発生
- アップロードが即時中止

![alt text](</img/スクリーンショット 2026-02-14 135028.png>)

<br>

---

### 結果

通信を強制的に停止させるには、

> ルーティングを不正状態にし、対向先への経路を消す

ことが有効であると確認できた。

## 補足事項

- 本検証では単一VNet構成で実施（Azure Firewallなし）
- Hub & Spoke構成の場合は、ルートテーブルの関連付け解除で遮断可能
- 自動化する場合は以下の構成が考えられる

<br>

---

### 自動化イメージ

1. ExpressRouteメトリック（帯域使用率／パケットドロップ）を監視  
2. 閾値超過でメトリックアラートを発火  
3. Automation Runbookを起動  
4. VNetフローログから高トラフィック宛先を特定  
5. 該当サブネットのルートテーブル関連付けを解除  

---

### 重要な制約事項

対向（OCI）側からのイングレストラフィックについては、

- ExpressRouteを通過する段階では制御不可
- GatewaySubnetのルートを外しても受信自体は防げない可能性が高い

つまり、Azure側のみで**完全な帯域占有防止は困難**である。

---

### Microsoft側の制御について
Microsoftが、ExpressRoute回線の流量制御を実施する旨を通達する記事が存在する。<br>
これは、帯域使用率が100%を超えた際にパケットを破棄する仕組みである。<br>
https://jpaztech.github.io/blog/network/expressroute_MV1F-T9G/<br>
<br>
もともとExpressRouteは冗長化の目的で2回線が標準使用され、1つの回線の帯域がオーバーしても、もう一方ヘバーストする。<br>
しかし、一部の利用者がこれを前提としてた帯域オーバーを起こしていることからこの流量制御が行われることになった。<br>
ExpressRoute回線におけるパケットのドロップは、これによって発生する。

---


## 結論

- 帯域を占有しても、他通信が完全停止するわけではない
- AzureにはExpressRoute帯域を直接制御する仕組みは存在しない
- 強制停止するにはルーティング無効化が有効だが、影響範囲が大きく、Microsoftのベストプラクティスにも存在しない
- ExpressRouteはスケーラブルなリソースではないため、段階的な帯域増強が必要
- 日常的な帯域監視およびメトリックアラート設定が重要

---

## 総括

ExpressRoute回線の帯域占有は、

- 即時に他の通信の全面停止には直結しない
- しかし通信品質が低下する可能性は否定できないため、帯域を100%近くまでに使うことは推奨されない。
    - 今回の検証も、帯域占有において最も有効と考えられる方法を選んだが、それでも帯域を**完全に占有すること**は不可能だった。
    - 100%以上を常にキープするわけではなく、時に下回ることがあるため、ファイルの欠損は再現できなかった。
    - ExpressRoutegatewayは弾力性がないものの、このわずかに余力のある帯域幅を利用し、ファイル転送ができたと考えられる。
    - この要因については、利用者の目に見えないボトルネック要素が複雑多様に絡み合うことも考えられるため、結論として特定は不可能である。
        - 仮想マシンのSKU
        - MicrosoftのDC内の物理サーバー、および物理回線のスペック
        - Microsoftの流量制御の有無　など

したがって、

**事前の監視設計と適切な帯域設計を行い、段階的に帯域の増速を行うことが望ましい**
